# AI Widget Platform – Dashboard PRD (Frontend)

## 1. Overview

This document defines the **Product Requirements for the Dashboard Application** of the AI Widget Platform.

The dashboard is a **frontend-only application** used by clients to manage their AI widgets.

The dashboard communicates with the backend API built with:

- Bun
- ElysiaJS
- PostgreSQL
- OpenRouter AI models

The dashboard must **not contain backend logic**. All business logic must live in the backend API.

The dashboard is responsible for:

- widget management
- context editing
- embed script retrieval
- account management

# 3. Design System Rules

The dashboard must strictly follow the **shadcn UI design system**.

Rules:

- Do not modify the design tokens generated by shadcn
- Use the `globals.css` file created by shadcn
- Use shadcn components whenever possible
- Maintain visual consistency across the dashboard

Allowed components:

- Button
- Input
- Textarea
- Dialog
- Card
- Table
- Dropdown
- Tabs
- Alert
- Badge

Custom components must **extend shadcn components**, not replace them.

The dashboard must follow strict engineering standards.

Authentication is handled by the backend.

The dashboard must:

- redirect unauthenticated users
- attach authentication tokens to API requests

- Google Sign-In

Future providers may include:

- email/password
- GitHub
- additional OAuth providers

# 8. Core Features

## 8.1 Dashboard Home

Purpose:

Provide a quick overview of the user's account.

Content may include:

- number of widgets
- quick actions
- recent activity (future)

---

## 8.2 Widgets Management

Users must be able to:

- create widgets
- edit widgets
- delete widgets
- view widget configuration

Widget fields:

- name
- context
- created_at
- updated_at

---

## 8.3 Widget Context Editor

Each widget contains a **context text area** where users define information used by the AI.

Examples of context:

- business description
- products
- services
- FAQs
- instructions for the AI

Requirements:

- large textarea
- save button
- clear editing UI
- validation

---

## 8.4 Widget Embed Script

Users must be able to copy the widget embed script.

Example:

<script src="https://cdn.example.com/widget.js" data-widget-id="widget_id"> </script>

Requirements:

- copy button
- readonly code block
- formatted display

---

## 8.5 Account Settings

Users must be able to:

- view profile information
- log out

Future additions:

- API keys
- billing
- usage metrics

---

# 9. API Communication

All API requests must go through a centralized API client.

Example structure:


services
api-client.ts
widgets-service.ts
auth-service.ts


Rules:

- no direct fetch calls inside components
- centralized error handling
- typed responses

---

# 10. State Management

Global state should remain minimal.

Allowed approaches:

- React state
- React context
- lightweight stores if necessary

Avoid large global state systems unless required.
